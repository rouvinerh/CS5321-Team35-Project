Blocking Cross-Session Token Splicing

This project studies a delegation flaw in a vulnerable STS and demonstrates how a session-bound capability chain, inspired by SIFF, changes the outcome from forged acceptance to rejection.
Threat: A stolen delegated token can be spliced with an attacker-controlled actor token.
Observation: Token validity alone is not enough. The delegation path must also be validated.
Contribution: Enforce path integrity, session binding, signed hop chains, and replay-safe delegation.

Background

OAuth, Delegation, and the STS

What is OAuth?

OAuth lets software act on a user's behalf without sharing their password via tokens: signed, time-limited passes that specify exactly what is permitted, for how long, and on whose behalf.

What is Delegation?

When a service needs to pass authority further down a chain, each handoff is a delegation hop. Permissions can only narrow at each step.

Example: A user authorises an AI assistant to read their inbox. The agent delegates a narrower credential to a sub-agent that can only schedule meetings.

Where it is used

  • AI agent pipelines acting on behalf of a user across multiple services
  • Cloud microservices calling each other under a delegated identity (AWS, Azure)
  • CI/CD runners acting on behalf of engineers in automated pipelines
The Security Token Service (STS) is the central authority that issues and validates tokens at each hop.

Background

Token Exchange

Subject Token

Represents the original user's authority. It declares who authorised the chain and what they permitted.

sub:  the original user (e.g. Alice)
aud:  the agent this token was issued to
scope:  the permissions granted at this hop

Actor Token

Proves the identity of the agent making the request right now. It declares who is presenting the subject token.

sub:  the agent currently acting
act:  records the chain of prior actors
scope:  the narrowed permissions being requested
The STS must verify that subject_token.aud matches actor_token.sub. This confirms the subject token was actually issued to the agent now presenting it, not just that both tokens are individually valid.

Problem

Why Token Exchange Needs Path Integrity

Core invariant
subject_token.aud == actor_token.act.sub

System Model

  • A user delegates through multiple agents via a shared STS.
  • JWTs carry sub, aud, act, and scope.
  • Each downstream exchange must preserve a continuous delegation path.

Vulnerability

  • The vulnerable STS verifies each token on its own.
  • It checks caller identity, but not whether the subject token was actually issued to that caller.
  • This gap enables cross-session token splicing.

Security Goal

  • Reject any exchange that joins tokens from unrelated sessions.
  • Prevent scope expansion and replay.
  • Keep the mechanism lightweight enough for short delegation chains.
The attack does not break cryptography. It exploits a missing path-validation step inside the exchange service.

Baseline

Normal Delegation Flow

Alice authenticates to Agent A.
The STS issues an initial token scoped to Agent A.
Agent A exchanges it for a narrower token scoped to Agent B.
This exchange is valid because the token was issued to the agent now presenting it. The path is continuous.
Figure: normal service-to-service delegation. Each token recipient matches the caller at that step.

Attack

Cross-Session Token Splicing

The attacker steals Agent B's delegated token.
It is paired with the attacker's own actor token and both are submitted to the vulnerable STS.
The caller check passes. The path check does not exist.
The STS mints a forged token, accepting a delegation path that never happened.
Figure: a stolen subject token and an attacker-controlled actor token are spliced into a single forged exchange request.

Defence

Session-Bound Capability Chain

A path-integrity check rejects any caller that was not the intended recipient of the subject token.
Each delegation hop is recorded as an STS-signed capability tied to a shared session identifier.
The secure STS validates signatures, session continuity, hop order, nonce freshness, and scope monotonicity.
The same attack input now fails before any new token is minted.
Figure: the secure validation pipeline. The attack is stopped at path integrity before minting begins.

Defence

Relation to SIFF

SIFF (Network Layer)

  • Routers issue capability tokens to legitimate flows
  • Packets without a valid capability for that flow are dropped
  • Path membership is enforced without storing per-flow state

This Project (Identity Layer)

  • The STS issues signed capabilities at each delegation hop
  • Exchange requests without a valid capability chain are rejected
  • Path membership is enforced without storing per-session state

Component Mapping

  • Flow identifier: Shared session identifier
  • Per-packet capability: Per-hop signed capability
  • Router capability check: STS path-integrity check
SIFF asks: Did this packet travel a legitimate authorised path? This project asks a similar question one layer up: Did the token travel a legitimate authorised delegation chain?

Result

Same Attack Input, Different Execution Result

Vulnerable STS

Forged token accepted

The caller check passes.

The path check is absent.

A new token is minted for the attacker, creating a delegation path that never existed.

Secure STS

Request rejected

The same attack input enters the secure exchange path.

Path integrity and session validation reject it before minting.

The delegation chain is treated as a security object, not just metadata.

Replay

Blocked by nonce freshness checks on the latest capability in the chain.

Scope Escalation

Blocked by requiring the requested scope to be a strict subset of the previous hop.

Tampering

Blocked by HMAC-protected capability signatures and chain continuity checks.

Conclusion

What the Demo Establishes

1. Real Vulnerability

The vulnerable STS can be driven into minting a forged token using only valid, properly signed inputs.

2. Real Change in Code

The secure design changes the execution result for the same attack from acceptance to rejection.

3. Manageable Cost

The defence adds sub-millisecond overhead that scales linearly. It is practical for real delegation chains.

The central claim is not just that this attack can be explained. It is that delegation integrity can be enforced in code, with a clear and practical validation path.
