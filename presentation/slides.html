Blocking Cross-Session Token Splicing in Token Exchange

Final Project Presentation
OAuth token exchange, delegation integrity, capability chains
We study a delegation flaw in a vulnerable STS and show how a session-bound capability chain changes the execution result from forged acceptance to rejection.
Threat: a stolen delegated token can be spliced with an attacker-controlled actor token.
Observation: token validity alone is not enough; the delegation path must also be continuous.
Contribution: enforce path integrity, session binding, signed hop chains, and replay-safe delegation.

Problem

Why Token Exchange Needs Path Integrity

Core invariant
subject_token.aud == actor_token.act.sub

System Model

  • User delegates through multiple agents using a shared STS.
  • JWTs carry sub, aud, act, and scope.
  • Each downstream exchange should preserve a continuous delegation path.

Vulnerability

  • The vulnerable STS verifies tokens individually.
  • It checks caller identity, but not whether the stolen token was issued to that caller.
  • This enables cross-session token splicing.

Security Goal

  • Reject any exchange that joins unrelated sessions.
  • Prevent scope expansion and replay.
  • Keep the mechanism lightweight enough for short delegation chains.
The attack does not break cryptography. It exploits missing delegation-path validation inside the exchange service.

Baseline

Normal Delegation Flow

Legitimate path
Alice → Agent A → Agent B
Alice first authenticates to Agent A.
The STS issues an initial token for Agent A.
Agent A delegates to Agent B with narrower scope.
This is a valid exchange because the token presented to the STS was actually issued to the current actor.
Figure: baseline service-to-service delegation. The path is continuous and the token recipient matches the current caller.

Attack

Cross-Session Token Splicing

Same attack input
different system outcomes
The attacker steals Agent B’s delegated token.
They combine it with their own actor token and submit both to the vulnerable STS.
The caller check passes, but the path check is missing.
The vulnerable STS mints a forged token and accepts a path that never existed.
Figure: a stolen subject token and an attacker-controlled actor token are spliced into one forged exchange request.
Old system end-to-end
Defense

Session-Bound Capability Chain

Secure STS checks
caller, path, session, chain, scope, replay
A direct path-integrity check rejects a caller that was never the intended recipient of the token.
Each delegation hop is recorded as an STS-signed capability with a shared session identifier.
The secure STS validates signatures, session continuity, hop order, path continuity, nonce freshness, and scope monotonicity.
The same attack input now fails before a new token can be minted.
Figure: validation pipeline in the secure design. The attack is stopped when path integrity and session consistency are enforced.

Result

Same Attack Input, Different Execution Result

Core comparison
vulnerable accepts, secure rejects

Vulnerable STS

Forged token accepted

The caller check passes.

The path check is missing.

A new token is minted for the attacker, creating a forged path.

Secure STS

Request rejected

The same kind of attack input reaches the secure exchange path.

Path integrity and session validation reject the request before minting.

The delegation chain is preserved as a security object, not just metadata.

Replay

Blocked by nonce freshness checks on the latest capability.

Scope Escalation

Blocked by requiring the requested scope to remain a subset of the previous hop.

Tampering

Blocked by HMAC-protected capability signatures and chain continuity checks.

Conclusion

What the Demo Establishes

Final takeaways
vulnerability, fix, cost

1. Real Vulnerability

The vulnerable STS can be driven into minting a forged token using only valid, signed inputs.

2. Real Change in Code

Our secure design changes the execution result for the same attack pattern from acceptance to rejection.

3. Manageable Cost

The defense adds measurable but still small overhead for short delegation chains.

The project’s central claim is not just that the attack can be explained, but that delegation integrity can be enforced in code with a clear and practical validation path.
